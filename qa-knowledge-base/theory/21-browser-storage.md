# 21. Работа браузерных хранилищ: Где сайт прячет данные

Когда ты закрываешь вкладку браузера, сайт не всегда «забывает» о тебе. Для этого используются механизмы клиентского хранения. Тестировщику важно знать, где лежат эти данные, чтобы проверять сценарии авторизации, кэширования и безопасности.

## 1. Cookies (Куки)
Самый старый и важный механизм. Это небольшие текстовые файлы, которые сервер отправляет браузеру через заголовок `Set-Cookie`.

### Ключевые атрибуты для проверки (QA-чеклист):
* **Domain & Path:** Определяют, для каких сайтов и страниц кука будет доступна. Ошибка здесь может привести к тому, что авторизация «отвалится» на поддомене.
* **Expires / Max-Age:** Определяет срок жизни. Если дата в прошлом — кука удаляется. Проверь, что сессия не «протухает» слишком быстро.
* **Secure флаг:** Если установлен, кука передается только по защищенному протоколу HTTPS.
* **HttpOnly флаг:** Критически важен для безопасности. Запрещает доступ к куке через JavaScript (`document.cookie`). Это главная защита от кражи сессии через XSS-атаки.
* **SameSite (Strict, Lax, None):** Регулирует, отправляется ли кука при переходах с других сайтов. Защита от CSRF-атак.

**Что тестировать:**
1. **Удаление:** Удали куку авторизации вручную — сайт должен мгновенно потребовать логин.
2. **Подмена:** Попробуй изменить значение куки, если оно не зашифровано.
3. **Очистка по таймауту:** Убедись, что после истечения срока жизни куки пользователя корректно разлогинивает.

## 2. Web Storage: LocalStorage и SessionStorage
Это современные хранилища (ключ-значение), которые работают только на стороне клиента и не передаются на сервер автоматически.

### LocalStorage (Постоянное хранилище):
* **Объем:** До 5-10 МБ (в зависимости от браузера).
* **Срок жизни:** Данные хранятся, пока пользователь не очистит кэш или скрипт не выполнит `localStorage.clear()`.
* **Примеры:** Список «Избранное», состояние темной темы, неотправленные черновики сообщений.

### SessionStorage (Сессионное хранилище):
* **Объем:** Аналогично LocalStorage.
* **Срок жизни:** Только до закрытия вкладки. Дублирование вкладки копирует данные, но открытие новой — создает чистое хранилище.
* **Примеры:** Шаги многостраничной формы регистрации, фильтры в каталоге, примененные только на время текущего поиска.

## 3. Сравнение хранилищ (Таблица для QA)

| Характеристика | Cookies | LocalStorage | SessionStorage |
| :--- | :--- | :--- | :--- |
| **Объем** | Очень мал (~4 КБ) | Большой (~5-10 МБ) | Большой (~5-10 МБ) |
| **Передача на сервер** | С каждым HTTP-запросом | Никогда | Никогда |
| **Доступ к JS** | Да (если нет HttpOnly) | Да | Да |
| **Срок жизни** | Устанавливается сервером | Бессрочно | До закрытия вкладки |

## 4. IndexedDB (Встроенная БД)
Это полноценная объектно-ориентированная база данных внутри браузера. Она сложнее, чем просто «ключ-значение».

* **Зачем QA это знать:** Используется в тяжелых Web Apps (PWA), которые должны работать без интернета. Например, Google Docs или Figma хранят здесь структуру документов.
* **Как тестировать:** В DevTools на вкладке Application можно увидеть таблицы и записи. Проверь, обновляются ли данные в таблицах при совершении действий на странице в оффлайн-режиме.

## 5. Практические манипуляции в DevTools
Открой **DevTools -> Application -> Storage**.

**Сценарии для глубокого тестирования:**
1. **Race Condition (Состояние гонки):** Что будет, если открыть две вкладки одного сайта и в одной изменить настройки (записав в LocalStorage), а во второй — попытаться их прочитать?
2. **Безопасность (Insecure Data):** Проверь, не хранит ли разработчик в LocalStorage чувствительные данные (пароли, номера карт, личные данные) в открытом виде. Это грубое нарушение.
3. **Лимиты:** Попробуй через консоль забить LocalStorage до упора (циклом). Как поведет себя сайт, когда место закончится? Он должен выдать понятную ошибку, а не просто «упасть».
4. **Манипуляция состоянием:** Измени в LocalStorage количество товаров в корзине или статус подписки. Посмотри, «поверит» ли фронтенд этим данным (если да — это баг валидации на стороне клиента).
