# 14. API и техническая документация (Swagger / OpenAPI)

Тестирование бэкенда начинается не с кнопок, а с понимания «контракта» между системами. Основной документ для этого — спецификация API.

## 1. Что такое API и зачем его тестировать?
**API (Application Programming Interface)** — это интерфейс, который позволяет одной программе (например, твоему браузеру) общаться с другой (сервером). 

**Роль QA в тестировании бэкенда:**
* Убедиться, что сервер обрабатывает данные согласно логике.
* Проверить безопасность (нельзя ли получить чужие данные).
* Подтвердить отказоустойчивость (что сервер не «упадет» от некорректного запроса).

## 2. Swagger (OpenAPI) — Интерактивная документация
**Swagger** — это самый популярный инструмент для визуализации API. Он читает код бэкенда и строит страницу, где можно изучить и сразу протестировать эндпоинты.



### Как читать документацию в Swagger:
1.  **Endpoint (URL):** Путь к ресурсу (например, `/api/v1/auth/login`).
2.  **Method:** HTTP-метод (GET, POST и т.д.).
3.  **Parameters:** Список входных данных. Бывают обязательные (Required) и опциональные.
4.  **Request Body:** Формат JSON-объекта, который нужно отправить.
5.  **Responses:** Список кодов ответов (200, 400, 500) и описание того, что они значат в этом конкретном методе.

## 3. Практические шаги тестирования в Swagger
Не ограничивайся только «позитивным» сценарием. Используй кнопку **"Try it out"** для следующих проверок:

* **Валидация полей:** Попробуй отправить строку в поле, где ожидается число (`integer`).
* **Обязательные поля:** Удали из запроса поле, помеченное как `required`, и проверь, вернет ли сервер ошибку `400 Bad Request`.
* **Граничные значения:** Если в документации указано, что `age` должен быть от 18 до 100, проверь, как сервер отреагирует на `17` или `101`.
* **Структура ответа:** Сравни JSON из ответа сервера с тем, что описано в разделе **Schemas** (внизу страницы Swagger). Все ли поля на месте? Совпадают ли их типы?



## 4. Ограничения Swagger
Хотя Swagger удобен для быстрой проверки, у него есть минусы для полноценного QA:
1.  **Нет автоматизации:** Нельзя запустить сразу 100 тестов одной кнопкой.
2.  **Нет сценариев:** Трудно тестировать цепочки (например: «Создать заказ» -> «Получить его ID» -> «Оплатить его»).
3.  **Зависимость от разработчика:** Если программист забыл обновить документацию в коде, Swagger будет врать.

*Для решения этих проблем мы используем **Postman**.*
