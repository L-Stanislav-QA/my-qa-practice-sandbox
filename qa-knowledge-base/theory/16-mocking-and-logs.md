# 16. Изоляция через Моки и анализ Логов

Тестировщику бэкенда важно не только уметь отправлять запросы, но и уметь имитировать ответы (Mocks), а также находить причины ошибок в «бортовых самописцах» сервера (Logs).

## 1. Mock-серверы (Заглушки)
**Mock (Мок)** — это имитация реального сервиса, которая возвращает заранее заданные данные.

### Зачем нужны моки в QA?
1. **Разработка фронтенда:** Бэкенд еще не написан, но фронтенду нужно на чем-то рисовать интерфейс. Мы создаем мок, который «прикидывается» рабочим API.
2. **Изоляция:** Нужно проверить, как приложение обрабатывает ошибку `500` или `403`. Реальный сервер ломать нельзя, а мок может выдать любой статус-код по запросу.
3. **Тестирование сторонних сервисов:** Если твое приложение интегрировано с платежной системой (например, любой банк), ты не будешь тратить реальные деньги при каждом тесте. Ты используешь мок платежки.

### Инструменты:
* **Postman Mock Server:** Позволяет быстро создать эндпоинт из коллекции.
* **Charles / Fiddler:** Можно «подменить» ответ от сервера на лету прямо в браузере.



## 2. Логирование (Logging)
**Логи** — это текстовые записи о том, что происходило внутри системы. Когда API возвращает `500 Internal Server Error`, ответ не говорит, *что именно* сломалось. Ответ кроется в логах.

### Где искать логи?
* **Логи приложения:** Ошибки в коде (Java, JS, Python).
* **Логи сервера (Nginx/Apache):** Ошибки доступа, проблемы с сертификатами.
* **Логи базы данных:** Медленные запросы, ошибки коннекта.

### Как читать логи (пример):

[2026-04-27 21:05:12] ERROR: NullPointerException at UserService.java:42
RequestID: abc-123 | UserID: 99

**На что смотреть:**
1. **Timestamp:** Время ошибки (должно совпадать с твоим тестом).
2. **Log Level:** 
    * INFO (просто инфо), 
    * WARN (предупреждение), 
    * ERROR (ошибка), 
    * FATAL (всё упало).
3. **Trace ID:** Уникальный номер запроса, по которому можно отследить его путь через все микросервисы.
4. **Stack Trace:** Список строк кода, которые привели к падению.

## 3. Практические советы для QA
* **Сначала логи, потом баг-репорт:** Перед тем как заводить баг на бэкенд, загляни в логи. Приложенный скриншот лога сэкономит разработчику час времени.
* **Использование `grep`:** В консоли сервера часто используют команду `grep "ERROR"` или `grep "RequestID"`, чтобы отфильтровать лишний шум и найти конкретное событие.
* **Проверка «молчаливых» ошибок:** Иногда API возвращает `200 OK`, но в логах пишется ошибка (например, не отправилось письмо). Это тоже баг.
