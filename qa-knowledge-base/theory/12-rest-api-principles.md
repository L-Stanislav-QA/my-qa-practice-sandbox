# 12. Принципы REST API

**REST (Representational State Transfer)** — это архитектурный стиль взаимодействия компонентов, в основе которого лежит управление ресурсами с помощью стандартных методов HTTP.

## 1. Основные принципы (Constraint)
Чтобы API считалось RESTful, оно должно соответствовать 6 правилам:

1.  **Клиент-серверная модель (Client-Server):** Разделение ответственности. Клиент занимается интерфейсом, сервер — данными.
2.  **Без сохранения состояния (Stateless):** Каждый запрос должен содержать всю информацию для его обработки. Сервер не «помнит», что ты присылал секунду назад.
3.  **Кешируемость (Cacheable):** Ответы сервера должны помечаться как кешируемые или нет, чтобы клиент не запрашивал одно и то же по многу раз.
4.  **Единообразие интерфейса (Uniform Interface):** Все запросы к разным ресурсам должны выглядеть похоже (использование URI, стандартных методов и форматов данных вроде JSON).
5.  **Многоуровневая система (Layered System):** Клиент не должен знать, общается он напрямую с сервером или через посредника (балансировщик, кэш).
6.  **Код по требованию (Code on Demand) (необязательно):** Сервер может передавать исполняемый код клиенту (например, JavaScript-скрипты).  Это единственное необязательное ограничение в архитектурном стиле REST, позволяющее серверу расширять функциональность клиента, передавая ему исполняемый код (например, скрипты JavaScript или апплеты). Это позволяет временно настраивать или дополнять возможности клиентского приложения без необходимости его полной переустановки



## 2. Ресурсы и URI
В REST всё является **ресурсом** (товар, пользователь, заказ). У каждого ресурса есть свой уникальный адрес — **URI (Uniform Resource Identifier)**.
* Правильно: `GET /users/123` (Получить пользователя с ID 123).
* Неправильно: `GET /getUserById?id=123` (Это стиль RPC, а не REST).

---
### Важное уточнение: URI, URL, URN
В контексте REST мы часто говорим об идентификации ресурсов. Нужно различать эти понятия:

* **URI (Uniform Resource Identifier)** — «Паспорт» ресурса. Это общая категория, которая просто уникально идентифицирует объект.
* **URL (Uniform Resource Locator)** — «Адрес» ресурса. Самый частый тип URI, который говорит, **как найти** объект (протокол + домен + путь).
* **URN (Uniform Resource Name)** — «Имя» ресурса. Уникальное название, которое не зависит от местоположения (встречается реже, например, `isbn:0451450523` для книги).

> **Аналогия:** 
> URI — это способ обозначить человека. 
> URN — это его ФИО (кто он). 
> URL — это его домашний адрес (где он живет).



#### Анатомия URL
`https://wildberries.com:443/catalog/item?id=123#reviews`
1. **https** — Протокол (Scheme).
2. **wildberries.com** — Домен (Host).
3. **443** — Порт (стандарт для HTTPS).
4. **/catalog/item** — Путь к ресурсу (Path).
5. **?id=123** — Параметры запроса (Query params).
6. **#reviews** — Якорь (Fragment).

## 3. Идемпотентность методов
Важное понятие для тестирования — что будет, если отправить запрос несколько раз?
* **Идемпотентные методы:** Повторный вызов дает тот же результат, что и первый (состояние сервера не меняется дальше).
    * `GET`, `HEAD`, `PUT`, `DELETE`.
* **Неидемпотентные методы:** Каждый новый вызов создает новый объект или меняет состояние.
    * `POST` (дважды нажали «Оплатить» — списалось дважды, если нет защиты).



## 4. Что проверять тестировщику в REST API?
1.  **Корректность эндпоинтов:** Соответствует ли URL ресурсу?
2.  **Коды ответов:** Возвращает ли сервер `201 Created` при создании и `404 Not Found`, если ресурса нет?
3.  **Валидация JSON:** Приходят ли все обязательные поля? Корректны ли типы данных (строка, число)?
4.  **Обработка ошибок:** Понятно ли сообщение об ошибке в теле ответа при `400 Bad Request`?
5.  **Идемпотентность:** Не создается ли дубликат заказа при повторном POST-запросе?
